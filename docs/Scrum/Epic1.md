1. Epic 1 — Repository & Solution Bootstrap
Цель: развернуть репозиторий и каркас решения так, чтобы дальше не было копипасты, а сборка/тесты/стиль были единообразными и автоматизированными.
Гейт после эпика: репозиторий собирается в CI, тесты гоняются, структура каталогов и базовые настройки зафиксированы.

1.1. Story 1.1 — Инициализация репозитория и базовая структура
Смысл: создать репозиторий с предсказуемой структурой и базовыми файлами управления.

Скоуп:
корневая структура каталогов
git-игнор
лицензия
базовые документы проекта

Выходные артефакты:
каталоги: src/, tests/, examples/, docs/, .github/workflows/
файлы: .gitignore, LICENSE, README.md (минимальный), CHANGELOG.md (пустой шаблон), CODE_OF_CONDUCT.md (опционально), CONTRIBUTING.md (черновик)
репозиторий создан/инициализирован

Acceptance Criteria:
В корне есть перечисленные каталоги и файлы, репозиторий чисто клонируется и открывается без ручной настройки путей.
.gitignore исключает bin/, obj/, .vs/, TestResults/, артефакты пакетов и временные файлы IDE.
LICENSE соответствует выбранной лицензии.

DoD:
Всё закоммичено одним/двумя коммитами с понятными сообщениями.
Структура соответствует договорённости и не требует ручной правки после клона.

Tasks:
T1.1.1 — Создать репозиторий и начальный коммит.
T1.1.2 — Создать каталоги src/, tests/, examples/, docs/, .github/workflows/.
T1.1.3 — Добавить .gitignore для .NET.
T1.1.4 — Добавить LICENSE.
T1.1.5 — Добавить минимальные README.md, CHANGELOG.md, CONTRIBUTING.md.

1.2. Story 1.2 — Создание решения и проектов (Core + тесты + пример)
Смысл: один .sln в корне, минимальный набор проектов, который уже компилируется и тестируется.

Скоуп:
Solution

тестовый проект
пример-консоль

Выходные артефакты:

Проекты добавлены в solution, корректные пути, нет “ручных” ссылок вне src//tests//examples/.

1.3. Story 1.3 — Единые сборочные настройки и качество кода
Смысл: единая политика компилятора и анализа кода во всех проектах, чтобы качество поддерживалось автоматически.

Скоуп:
nullable
warnings as errors (управляемо)
стиль/форматирование
общие свойства сборки

Выходные артефакты:
Directory.Build.props (в корне)
Directory.Build.targets (при необходимости)
.editorconfig (в корне)
настройки: Nullable=enable, TreatWarningsAsErrors=true (или строгое подмножество), LangVersion, AnalysisLevel, Deterministic=true

Acceptance Criteria:
Любой новый проект, добавленный в solution, автоматически наследует настройки.
dotnet build при нарушениях nullability/анализа падает (согласованный набор правил).
Форматирование фиксируется .editorconfig.

DoD:
Настройки не конфликтуют с netstandard2.0 (если он выбран) и тестовым net8.0.
Документировано в docs/ или в комментариях, почему выбран такой набор правил.

Tasks:
T1.3.1 — Добавить Directory.Build.props с общими свойствами (nullable, langversion, analysis).
T1.3.2 — Добавить .editorconfig (минимально: indent, charset, newline, csharp_style).
T1.3.3 — Включить analyzers (встроенные) и согласовать уровень строгости.
T1.3.4 — Прогнать сборку и устранить первые предупреждения/ошибки.

1.4. Story 1.4 — CI: сборка и тесты на push/PR
Смысл: автоматическая проверка, что репозиторий собирается и тесты проходят на каждом изменении.

Скоуп:
GitHub Actions workflow (или выбранная CI)
сборка + тесты
кеширование пакетов
матрица версий SDK (минимально одна)

Выходные артефакты:
.github/workflows/ci.yml (или эквивалент)
CI запускается на push и pull_request
шаги: restore → build → test

Acceptance Criteria:
При каждом push/PR workflow стартует.
В случае ошибок сборки/тестов CI падает.
В случае успеха CI зелёный.

DoD:
Используется actions/setup-dotnet с фиксированной версией (например, 8.0.x).
Добавлено кеширование NuGet (~/.nuget/packages), если это не ломает воспроизводимость.
Команда тестов включает --configuration Release (по договорённости) или явно описан режим.

Tasks:
T1.4.1 — Создать workflow ci.yml.
T1.4.2 — Настроить триггеры push/pull_request.
T1.4.3 — Добавить шаги dotnet restore, dotnet build, dotnet test.
T1.4.4 — Добавить кеш NuGet (опционально).
T1.4.5 — Проверить на PR, что workflow реально отрабатывает.

1.6. Story 1.6 — Заготовка под NuGet-публикацию (метаданные без релиза)
Смысл: подготовить проекты так, чтобы когда дойдём до релиза, не пришлось массово переделывать csproj.

Скоуп:
csproj метаданные пакета (без публикации)
версионирование (без выбора конкретного процесса релизов)

Выходные артефакты:
в Uncertainty.Core.csproj прописаны базовые поля: PackageId, Version (или VersionPrefix), Authors, RepositoryUrl, PackageLicenseExpression, Description, PackageTags
включено GeneratePackageOnBuild=false (пока), чтобы не плодить .nupkg на каждом билде

Acceptance Criteria:
dotnet pack -c Release отрабатывает локально и создаёт пакет.
Пакет содержит корректные метаданные и файлы.

DoD:
Метаданные не содержат выдуманных данных; если чего-то нет — ставится нейтральное временное значение или оставляется пустым до Epic релиза.

Tasks:

T1.6.2 — Проверить dotnet pack локально.
T1.6.3 — Добавить исключение .nupkg в .gitignore, если нужно.
